  HIGH     [privacy] The 'purge already refuses loudly on an untruncatable WAL' carve-out is false on the retry path, so `holds === false` blesses residual plaintext  conf 0.70
    .adlc/specs/11-erasure-tier-completeness.md:68-71
    The spec explicitly excludes WAL residue from `holds`'s question on the grounds that `purge` already covers it. It does not, on the one path that matters. `SqliteBackend.purge` gates the checkpoint on `if (removed > 0)` (sqlite.ts:215): it deletes the rows, then only attempts `wal_checkpoint(TRUNCATE)` when this call actually removed something. The failure mode the code itself documents (sqlite.ts:201-214, and erase.ts:241-244) is precisely 'the rows were deleted but the WAL could not be truncated ‚Äî retry'. On that retry the rows are already gone, so `removed === 0`, so NO checkpoint is attempted and no error is raised. `heal`'s sweep on the next boot has the same property (it passes the whole accumulated dead set to `purge`, which finds nothing to delete).
    
    Under the current code the retry silently succeeds too, so this is not introduced by T67 ‚Äî but T67 promotes `holds === false` to THE authoritative ¬ß11 completeness verdict and will land a `spec/` section stating that completeness is decided by byte-presence. `holds` as specified for sqlite is `SELECT 1 FROM deltas WHERE id = ?`, which is false while the pre-delete page images may still sit in `store.db-wal`. The ticket therefore ratifies a false completion in a document whose entire subject is false completions, and the premortem's seven scenarios do not include it.
    
    Honest bound: better-sqlite3's `close()` closes the last connection, which normally checkpoints and removes the `-wal` sidecar, so the residue is bounded by process lifetime rather than permanent. That does not rescue the claim ‚Äî the completion is REPORTED, and the compliance record written, while the bytes are still at rest.
    ‚úó failure: A second handle (a reader, a backup agent) holds a read transaction past `busy_timeout` during `erase(X)`. `purge` deletes the row and throws the INCOMPLETE-erasure error; `erase` rejects. The operator idles the other handle and re-runs `erase(X)` as instructed. `purge` now finds no row, returns 0, skips the TRUNCATE checkpoint entirely, `holds(X)` is false, and `erase` resolves `{ erased: X }`. `strings store.db-wal` still yields X's claims, and the store now has a recorded, verified-complete erasure over legible plaintext.
    ‚Üí fix: Either (a) drop the `removed > 0` gate in `SqliteBackend.purge` so the TRUNCATE checkpoint is attempted (and its `busy` result checked) whenever the driver is in WAL mode ‚Äî the retry must be able to finish the work the first attempt started; or (b) extend `holds` for sqlite to consult WAL state as well as the row, and delete this carve-out. Whichever is chosen, add an acceptance criterion that a second `erase` attempt after a failed WAL truncation still refuses until the WAL is actually truncated ‚Äî that is the rail this carve-out currently removes.
