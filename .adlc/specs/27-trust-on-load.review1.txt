The blessing model is thoughtfully argued, but three security-relevant guards it leans on are under-specified in ways a correct-looking implementation would build wrong: the root-name guard is a check-then-publish with no stated atomicity (TOCTOU under concurrent adoption re-opens the exact confused-deputy it defends against), the pen/law-kind classification that gates blessAll has no stated source of truth (bypassable if read from a self-declared manifest), and the authoritative `lawFrom` address-intersection over-attributes shared content across modules.   [31mHIGH    [39m [35m[concurrency][39m [1mRoot-name guard is check-then-publish with no stated atomicity — concurrent adoptions re-open the confused-deputy it exists to prevent[22m  [2mconf 0.60[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    The headline guard is specified as a read (does the root already register this semantic name with different content?) followed by an ordinary publish. Nothing in the artifact states that the check and the resulting registration are atomic, ordered, or serialized against other adoptions or direct publishes. §21 is described as latest-wins living semantics, so two guard evaluations that both observe 'no different-content Post yet' will both proceed. The spec explicitly grounds the whole guard in the claim that 'the hub world guarantees the collision arrives ... then as a one-line confused-deputy attack' — yet a racing pair of adoptions bypasses the guard entirely, and no acceptance criterion exercises concurrent or interleaved adoption.
    [33m✗ failure:[39m Operator (or a script) adopts module A's `Post` and module B's `Post` near-simultaneously; both are different-content and neither yet exists at root. Both guard reads see an empty `Post` name, both pass, both publish. Latest-wins silently installs whichever landed second as root `Post`, capturing every subsequent root read through un-reviewed law — the precise confused-deputy the guard was added to stop, now reachable without `supersede`.
    [32m→ fix:[39m State the guard's concurrency contract: the name-existence check and the registration must be atomic (e.g. a compare-and-set against the currently-winning registration for that semantic name, or a serialization point for adoptions to the same name), and add an acceptance criterion for two concurrent adoptions of different-content same-name law resolving to exactly one refusal.

  [31mHIGH    [39m [35m[security][39m [1mblessAll's pen/law-kind gating has no specified source of truth — a hostile manifest can misclassify a pen-holding renderer past the guard[22m  [2mconf 0.55[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    Two security-relevant branches — 'enumerate what it is about to bless BY LAW KIND' and 'a pen-holding renderer (§23.3) never rides the sugar ... refused otherwise' — depend on correctly classifying each row's kind and pen-ness. The module under adoption is repeatedly described as a stranger's, and the blessing is 'resolved through the manifest.' The artifact specifies that the *mirroring* claim is verified by address equality, but never states that a row's LAW KIND or PEN-ness is derived from the verified content-addressed export rather than a self-declared, adversary-controlled manifest field. If classification is read from the manifest, the guard is bypassable by a lying manifest.
    [33m✗ failure:[39m A hostile module ships a manifest that labels its pen-holding renderer as an ordinary schema row (or omits the pen bit). The operator runs `blessAll` without the distinct pen flag; the enumeration-by-law-kind and the pen guard both classify from the manifest, so the pen-holding renderer rides the sugar into the root trust domain — exactly the 'renderer-with-a-pen dragged into the root' outcome the design calls a footgun.
    [32m→ fix:[39m Require that law kind and pen-ness are read from the verified content-addressed export (or an authenticated/signed manifest whose claims are checked against the export bytes), never from an unauthenticated manifest field, and add an acceptance criterion where a manifest that misdeclares a pen-holding renderer is still refused by blessAll.

  [33mMEDIUM  [39m [35m[correctness][39m [1m`lawFrom` address intersection over-attributes law shared by multiple modules[22m  [2mconf 0.60[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    The spec makes content-address intersection between currently-bound law and a module's manifest the single authoritative answer to 'what law of module M binds in my root?' and motivates a future 'revoke-by-module' tool. Because law is content-addressed, two distinct modules that both list an identical standard schema (a common `Post`) share the same address. `lawFrom(M)` and `lawFrom(N)` will both report that row, with no way to tell 'M is where I got it' from 'M merely also lists it.' The witness-vs-adopted-from distinction guards the RECORD ledger against this, but `lawFrom` is defined as address arithmetic that ignores records — so the authoritative query inherits the over-attribution the records were fixed to avoid.
    [33m✗ failure:[39m Root has adopted a common `Post` schema from module A; module B's manifest also lists that identical schema. A future revoke-by-module tool built on `lawFrom(B)` (the query the spec blesses as authoritative) reports the shared `Post` as 'law from B' and revokes it, breaking A's — and the root's — resolution, even though B was never its source.
    [32m→ fix:[39m Specify `lawFrom`'s semantics for content shared across modules — either define it explicitly as 'law this root binds that M's manifest also lists' (attribution-agnostic, and say so, so no revoke tool treats it as provenance) or intersect it with adoption/witness records so shared law is not silently attributed as originating from every lister.

  [33mMEDIUM  [39m [35m[other][39m [1mRoot-name guard's 'different content' and `supersede`'s effect on the prior registration are ambiguous under §21 latest-wins[22m  [2mconf 0.55[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    The guard refuses when the name 'already carries a root registration with different content,' but §21 is latest-wins living semantics where a name accretes registrations over time. It is unspecified whether 'a root registration' means the currently-winning one or any historical one — so a module's `Post` matching a superseded-but-present registration is 'same content' (no-op) under one reading and 'different content' (refuse) under another. Separately, `supersede` is defined only as 'take over the living name, eyes open' with no statement of what happens to the prior registration (negated, or merely outranked and able to resurface if the new one is later negated). Two competent engineers will build both behaviors differently, with security-relevant divergence.
    [33m✗ failure:[39m Engineer A implements the guard against the currently-winning registration only; Engineer B against any historical registration. On the same store, adopting a module `Post` that equals an older superseded `Post` is an idempotent no-op for A and a hard refusal for B — divergent trust outcomes from identical inputs. Meanwhile a module that later negates its own `supersede`-installed `Post` either resurfaces the operator's original law (A) or leaves the name unresolved (B), unspecified by the artifact.
    [32m→ fix:[39m Define 'different content' against the currently-winning registration explicitly, and specify `supersede`'s effect on the prior registration (negate vs. outrank) plus what a reader resolves after a superseding registration is itself later negated; add object-level acceptance criteria for both.
