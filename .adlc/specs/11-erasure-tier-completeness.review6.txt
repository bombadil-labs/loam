 NEEDS ATTENTION   branch t67-tier-blind-erasure vs main (merge-base 060d7a2ec780b58fbd23a24b4d97e88ed86b7283)

Summary
  Do not land as-is. The core §11 fix is real — `holds` is a genuine byte-presence question, `MirrorBackend.holds` probes both tiers and fails closed, the pool fan-out is settled and claims `seen` synchronously, and the retry anchor is bounded by outstanding work. But the retry guard and the completion verdict now disagree about what a fault is: the verdict rejects on tombstone-delivery failure and purge refusal, while the guard asks only about BYTES — so an erase rejected by a pool that never held the id can never be completed through `erase` again, which is the exact stranding class this ticket was written to delete, moved one level out. Two smaller defects ride along: `await gw.reseat()` sits unguarded between the deliberately-caught local purge and the fan-out, so the same tier fault the catch exists to survive still aborts everything and leaves the gateway half-torn-down; and the durable sqlite truncation debt is never cleared by `close()`, the one operation that actually satisfies it, so a reopened handle reports retention over provably-absent bytes and a second `erase` resolves `{erased}` for no work.

  MEDIUM   [correctness] The retry guard asks only about bytes while the verdict also fails on tombstone delivery, so a pool that never held the id strands the erasure permanently  conf 0.65
    src/gateway/erase.ts:267-268
    The `nothing to erase` bypass is gated on `heldAnywhere`, which asks nothing but `gw.backend.holds(id)` on this gateway and, recursively, on each pool (erase.ts:370-381). The verdict that produces the rejection is strictly wider: `eraseReplicaImpl` throws when the operator's tombstone fails to LAND in a pool (`if (!readTombstones(gw.reactor, gw.operatorAuthor).has(id))`, erase.ts:452) regardless of whether that pool holds any bytes for the id, and `eraseImpl` unshifts a fault for a local `purge` that refused. A rejection caused by a non-byte fault is therefore unclearable: on the retry the primary has already purged and re-seated, so `target === undefined`, `already` is defined, and `heldAnywhere` answers false because nobody holds the bytes.
    
    A pool need not hold the id. `openQuarantineImpl` accepts `admit` and `membership` (quarantine-pool.ts:38-49) and its seeding edge admits the operator's tombstones unconditionally — 'A membership filter narrows what the pool SEES, never what it must FORGET' — so the fan-out visits every attached pool whether or not it was ever seeded with the target. A selectively-seeded pool whose backend refuses a write at fan-out time produces exactly this shape.
    
    The consequence is worse than a bad error message: the pool never receives the tombstone, so it keeps no record of the hole and its federation door will not refuse the id's re-entry, and no invocation of `erase` can ever deliver it. That is the failure the working spec describes as the reason the fix could not be one line, reappearing one level out. Inference note: I read the post-change source and traced the call order; I did not execute the suite, and the new fan-out rails all plant retention in pools that DO hold the target, so none of them reach this case.
    ✗ failure: An operator attaches a long-running quarantine pool P over a durable backend, seeded with a membership Term that excludes record X. `erase(X)` records the tombstone, purges X from the primary, and re-seats — X leaves the reactor. The fan-out calls `P.eraseReplica`, whose `federate` throws because P's store is momentarily unwritable. `erase` rejects with 'an attached quarantine pool refused'. The operator repairs P's store and re-runs `erase(X)` exactly as the error instructs: `already` is defined, but `heldAnywhere` finds the primary clean and P clean (P never held X), so the call rejects with `nothing to erase: X is not held here`. P has no tombstone for X for the rest of its life, the compliance record shows a failed erasure nobody can clear, and P's door will admit X back from any peer.
    → fix: Make the guard's fault model match the verdict's. Cheapest correct fix: let the bypass through whenever this erasure is OUTSTANDING for any reason the verdict can reject on — bytes held anywhere, OR a reachable pool that does not yet carry the operator's tombstone for `id` (`!readTombstones(pool.reactor, pool.operatorAuthor).has(id)`). Fold both into `heldAnywhere` (rename it to something like `erasureOutstanding`) so the two halves cannot drift again, exactly as `incompleteErasureFaults` is already shared by both fan-out layers. Add the missing rail beside the existing ones in test/gateway/erasure-fanout.test.ts: a pool opened with `admit: () => false` whose backend refuses during the fan-out, then repaired — the re-run must complete and the pool must end up holding the tombstone, not `nothing to erase`.

  MEDIUM   [error-handling] `gw.reseat()` is unguarded between the caught local purge and the fan-out, so the same tier fault still aborts every replica and leaves the gateway half-torn-down  conf 0.60
    src/gateway/erase.ts:310-316
    The local purge is now caught into `localPurge` with an explicit rationale: 'thrown here it would deny the tombstone and the sweep to every attached pool — one tier's fault becoming every replica's leak.' The very next statement, `await gw.reseat()` (erase.ts:316), is not guarded and reads from the same backend: `reseat` calls `this.backend.deltasSince(new Set())` (gateway.ts:632). Any fault that makes `purge` refuse and also makes a read refuse — a vault whose mount went read-only or EIO, a mirror whose primary tier faults — throws straight out of `eraseImpl`, so the pool fan-out at erase.ts:334 never runs and the collected `localPurge` fault is discarded. The mitigation therefore does not cover the class it was written for; it only covers backends that fail writes while still serving reads.
    
    Worse, `reseat` is destructive before it can fail: it returns every live channel, clears `sinks`, `lazyMats`, `publicLazyMats` and `resolverMemo`, and sets `publicOpen = undefined` (gateway.ts:622-628) BEFORE the `deltasSince` that can throw. A throw there leaves the gateway with every subscription killed, every cache cleared, `_reactor` still pointing at the pre-purge reactor, and `attachPersistence`/`rebindImpl` never re-run. Before this diff a refusing `purge` aborted ahead of `reseat`, so that torn state was unreachable on this path; catching the purge is what makes it reachable. Inference note: read from source, not executed; I did not audit whether every caller tolerates a gateway left in that state.
    ✗ failure: A store runs `MirrorBackend(archive, archive)` whose primary vault sits on a mount that flips read-only mid-operation. `erase(X)` appends and flushes the tombstone, then `gw.backend.purge([X])` throws EROFS — caught, as designed. `gw.reseat()` then returns every live channel and clears every cache, and its `deltasSince` throws EIO on the same mount. `erase` rejects with a reseat error that names neither the purge fault nor any tier; the two attached quarantine pools receive neither the tombstone nor a purge, so they keep X and stay able to re-admit it; and the caller is left holding a gateway whose subscribers have all been woken with `done` while its reactor still contains the record the store just promised to forget.
    → fix: Wrap `await gw.reseat()` the same way the purge is wrapped and push its rejection into the fault list, so a backend that cannot be re-read still gets the tombstone and the purge out to every pool before the call reports. Apply the identical change in `eraseReplicaImpl` (erase.ts:463). Independently, make `reseat` acquire the fresh delta set BEFORE it tears down channels and caches, so a failed read leaves the gateway untouched rather than half-reset. Rail it: a backend whose `purge` AND `deltasSince` both refuse must still leave every attached pool tombstoned and swept, and must leave the gateway usable.

  LOW      [correctness] The durable truncation debt is never cleared by `close()`, the one operation that satisfies it, so a reopened store reports retention over provably-absent bytes  conf 0.60
    src/store/sqlite.ts:305-320
    `SqliteBackend.holds` answers true for any id in `truncationOwed` (sqlite.ts:313), and that set is persisted to the `meta` table and re-read in the constructor (sqlite.ts:105-116). It is cleared in exactly one place: a successful `wal_checkpoint(TRUNCATE)` inside `purge` (sqlite.ts:263-265). `close()` is `this.db.close()` and nothing else (sqlite.ts:320-322) — but closing the last connection is precisely what checkpoints and removes the `-wal` sidecar, which the branch's own rail asserts ('close would checkpoint and settle the debt gracefully', test/store/erasure-at-rest.test.ts). So the operation that discharges the debt is the one operation that does not record it as discharged.
    
    After a busy checkpoint followed by a graceful close with no retry, the reopened handle carries a phantom debt: `holds(id)` returns true for an id whose bytes are gone. That is the safe direction for the §11 verdict, but it changes what the public API returns. `erase(id)` for that cleanly-erased id now finds a surviving tombstone, gets `heldAnywhere === true` from the phantom debt, takes the bypass, appends nothing, purges nothing (whose checkpoint then succeeds and silently clears the debt), finds no fault, and resolves `{ erased: id }` — a completion report for work never done. That is the case the new rail 'a second erase of a CLEANLY erased id still refuses' exists to pin, and the rail cannot see it because it runs on `MirrorBackend(MemoryBackend, MemoryBackend)`. Inference note: this rests on SQLite checkpointing and unlinking the WAL when the last connection closes (better-sqlite3's `close()` does not suppress that), which I did not verify by running the driver.
    ✗ failure: A backup agent holds a read transaction while `erase(A)` runs. The rows are deleted, `wal_checkpoint(TRUNCATE)` returns busy, `{A}` is written to the `meta` debt row, and `purge` throws; `erase` rejects. The operator ends the maintenance window and shuts the service down instead of retrying — `close()` checkpoints, the `-wal` file is removed, and A's plaintext is genuinely gone, but the debt row is left behind. On the next boot `holds(A)` reports true. A compliance script re-runs `erase(A)` to confirm the record is gone and receives `{ erased: A }` rather than the `nothing to erase` rejection it uses to classify records this store no longer holds, so the audit log records an erasure the run never performed.
    → fix: Discharge the debt where it is actually discharged: in `close()`, run `wal_checkpoint(TRUNCATE)` before `db.close()` and, on success, clear `truncationOwed` and delete the `meta` row — then a graceful shutdown leaves no phantom. As a belt-and-braces check, have the constructor verify the debt is still real (a `-wal` sidecar of non-zero size beside `filePath`) before trusting the row, and drop it otherwise. Extend the existing crash-boundary rail with its complement: fail the checkpoint, then `close()` normally, reopen, and assert `holds(id) === false` and that `erase(id)` rejects with `nothing to erase`.
