Summary
  The core §11 fix is real and well-railed — `holds` is a genuine byte-presence question, `MirrorBackend.holds` probes both tiers and fails closed with the tier named, the fan-out settles and claims `seen` synchronously, and the retry anchor is a surviving tombstone bounded by outstanding work. Two ride-along changes should not land as written. (1) `Gateway.reseat` was reordered to read-before-teardown, which moved the whole `channel.return()` await sequence between the snapshot and the reactor swap: a delta accepted during that window is written to the backend and then dropped from the live reactor, invisible until a restart — and a later `erase` of that id now answers `nothing to erase` over bytes that are still at rest, which is the exact hazard family this PR exists to close. (2) The sqlite truncation debt is the sole record that a purge left WAL truncation owed and that `holds` must answer unprovable, yet it is written to `meta` AFTER the DELETE commits and its failure is swallowed by a bare `catch {}` — so a failed write, or a crash in that window, silently reproduces the post-restart stranding the durable debt was introduced to eliminate. Fix both and re-prosecute; I could not execute the suite or the mutator, so no rail claim in this branch is independently confirmed here.

  MEDIUM   [correctness] reseat's read-before-teardown reorder drops any delta accepted during the channel teardown from the live reactor, while the backend keeps it  conf 0.50
    src/gateway/gateway.ts:625-643
    `reseat()` now snapshots the fresh reactor first (gateway.ts:625-631) and tears down afterwards, so the entire `for (const channel of [...this.channels]) await channel.return();` sequence (gateway.ts:635) plus the cache clears now sit BETWEEN the `deltasSince` read and `this._reactor = reactor` (gateway.ts:643). Before this diff the read was the last step, immediately followed by the swap with no intervening await.
    
    Any delta that is ingested during that window lands in the OLD reactor. `attachPersistence` is subscribed to that old reactor and is never unsubscribed, so the delta is still queued onto `this.writes` and written to the backend (gateway.ts:286-300) — but the new reactor was built before it existed, and the swap discards it. The result is an accepted, persisted delta that the live gateway cannot see: `gw.reactor.get(id)` is undefined, every registered View misses it, and nothing reports an error. It reappears only on the next `reseat()` or a reopen.
    
    The window is reachable in ordinary operation because `Channel.return()` resolves immediately (`Promise.resolve({value: undefined, done: true})`, channel.ts:72-74) — each `await` yields a microtask turn, and with `MemoryBackend` or better-sqlite3 (synchronous) a concurrent `gw.append()` resumes in exactly those turns. `erase` is the caller, so this fires on the compliance path.
    
    The reorder is defensible on its own terms (a failing `deltasSince` no longer leaves a half-torn-down gateway), but it is not the only way to get that property, and it was made without a rail: nothing in the diff asserts anything about a concurrent append across `reseat`. Inference note: I read the post-change source and traced the await points; I was not able to execute the suite in this environment, so the microtask ordering is reasoned rather than observed.
    ✗ failure: A gateway serves concurrent writes while the operator runs a subject erasure: `Promise.all([gw.erase(X), gw.append([D])])` over a MemoryBackend or sqlite store. `erase` purges X and calls `reseat`, which builds the fresh reactor from the backend. `D` has not been ingested yet. During the `await channel.return()` turns, `append`'s continuation runs: `D` is ingested into the old reactor and enqueued for the backend. `reseat` then swaps in the reactor built before `D` existed. `gw.append` resolves successfully, `D` is durably in the store, and the live gateway does not have it — reads omit it with no error. If a later subject request targets `D`, `eraseImpl` finds no target and, because no operator tombstone names `D`, throws `nothing to erase: D is not held here` while `D`'s bytes sit at rest in the backend — a false absence report on the erasure path.
    → fix: Keep the non-destructive-on-failure property without widening the window: read `const fresh = await this.backend.deltasSince(new Set())` first to prove the backend is answerable, then tear down channels and caches, then re-read (or re-ingest anything the old reactor accepted in between) immediately before `this._reactor = reactor` with no await in between. Alternatively, unsubscribe/quiesce writes for the duration — block `ingestVia` into a buffer that is drained into the new reactor after the swap. Either way add a rail: start a `gw.append` concurrently with `gw.erase` on a live subscription and assert the appended delta is present in `gw.reactor` after the erase resolves, and that `deltasSince` and the reactor agree.

  MEDIUM   [error-handling] The sqlite truncation debt is written after the delete commits and its failure is silently swallowed, so a lost debt row reproduces the post-restart stranding it was added to eliminate  conf 0.60
    src/store/sqlite.ts:270-283
    `truncationOwed` (sqlite.ts:58) is the only record that a purge deleted rows under a WAL it could not truncate, and this PR promotes it to a load-bearing input of the §11 verdict: `holds` returns true for any owed id (sqlite.ts:305-317), and that `true` is what lets `erasureOutstanding` admit the operator's retry past the `nothing to erase` guard (erase.ts:388).
    
    But the persistence of that record is neither atomic with the deletion nor checked. The DELETEs are committed at `this.db.exec("COMMIT")`, the checkpoint is attempted, and only then — in the `busy` branch — is the debt written, inside a bare `try { INSERT OR REPLACE INTO meta ... } catch { /* the loud throw below is the primary signal */ }` (sqlite.ts:274-282). Two paths lose it:
    
    1. The INSERT itself fails (disk full, `SQLITE_BUSY` on the write, a read-only mount) — the catch discards the error and the code proceeds to throw the INCOMPLETE-erasure message, which tells the operator to retry. Nothing anywhere records that the durable half of the obligation was not written.
    2. The process dies between the COMMIT and the INSERT — a window this code deliberately does not close, while the comment above the field says the debt is "AS DURABLE AS THE ERASURE IT BELONGS TO."
    
    In both cases the reopened handle reads no debt, `holds(id)` answers false over a `-wal` sidecar that may still carry the pre-delete page images, and the operator's retry — the remedy the error text prescribes — is refused as `nothing to erase` because `erasureOutstanding` finds nothing outstanding. That is precisely the stranding class the working spec says the durable debt was introduced to delete, merely narrowed to the unlucky window. The branch's crash rail in `test/store/erasure-at-rest.test.ts` exercises the path where the write DID land, so nothing pins either failure mode. Inference note: this is read from the post-change source; I could not execute the suite here, and better-sqlite3's `close()` normally checkpoints and unlinks the sidecar, which bounds the plaintext exposure to unclean termination — the stranding and the misleading rejection are unconditional.
    ✗ failure: A backup agent holds a read transaction. `erase(X)` records the tombstone, `purge([X])` DELETEs and COMMITs X's rows, `wal_checkpoint(TRUNCATE)` returns `busy: 1`, and the `INSERT OR REPLACE INTO meta` fails (the same contention, or a full disk) and is swallowed. `purge` throws INCOMPLETE, `eraseImpl` collects it, `holds(X)` is true from the in-memory set, and `erase` rejects telling the operator to resolve the fault and re-run. The host is then restarted (or the operator retries in a new process). The new handle finds no debt row, `holds(X)` is false, `erasureOutstanding` finds the tombstone delivered and no bytes anywhere, and `erase(X)` rejects with `nothing to erase: X is not held here`. No invocation of `erase` can drive the checkpoint the first attempt left owed, the compliance record shows a failed erasure nobody can clear, and X's pre-delete page images may still be legible in `store.db-wal` until an unrelated write triggers a checkpoint.
    → fix: Make the obligation as durable as the code claims. Write the debt row inside the SAME transaction as the DELETEs — record the candidate ids before COMMIT and delete the row afterwards only when the checkpoint actually succeeds — so a crash can only leave the debt over-recorded (the fail-closed direction), never under-recorded. Stop swallowing the write failure: if the debt cannot be persisted, say so in the thrown message (the operator must know the retry may not survive a restart) rather than reporting only the truncation fault. Add the two missing rails beside the existing crash rail: (a) make the `meta` write throw, abandon the handle without `close()`, reopen, and assert `erase(id)` is NOT refused as `nothing to erase`; (b) kill between COMMIT and the meta write (simulate by stubbing the INSERT) and assert the same.

  LOW      [resource-exhaustion] ArchiveBackend.holds falls through to a whole-vault cross-fan walk on every negative answer, and the erase path now asks it once per tier plus once per pool  conf 0.65
    src/store/archive.ts:238-263
    `ArchiveBackend.holds` (archive.ts:238) short-circuits only on a POSITIVE canonical-name hit; every absent answer lists every fan under the vault root and reads every filename in each, with a suffix test and string compare per name. That exhaustive reach is genuinely required — the straggler and misfiled-copy rails in `test/store/archive.test.ts` depend on it — but the absent case is the common one on the compliance path.
    
    This diff wires `holds` into three new call sites per erase: `incompleteErasureFaults` asks `gw.backend.holds(id)` on every erase (erase.ts:413), `eraseReplicaImpl` asks it again for every attached and nested pool, and the retry guard's `erasureOutstanding` walks the same set again before any of that (erase.ts:388). After a successful purge the canonical name is gone, so each of those is a full walk. Under `MirrorBackend(sqlite, archive)` a single erase now pays at least one whole-vault walk that did not exist before, on top of the walk `ArchiveBackend.purge` already performs, and a retry pays two.
    
    This is the repo's own H8 shape introduced on the erasure path. It is a cost finding, not a correctness one — the reach is right, and the code states the tradeoff ("Purges and probes are rare; the walk is cheap enough to be thorough") — but nothing in the diff measures it, and two prior reviews recommended a targeted-then-exhaustive ordering of which only the positive half was applied.
    ✗ failure: An operator replays a bulk subject-request script — a few thousand `erase(id)` calls — against a mirror whose cold tier is an `ArchiveBackend` holding ~100,000 archived deltas, with one quarantine pool attached. Every call that reaches the verdict performs a full cross-fan walk (~256 readdir syscalls plus ~100,000 filename comparisons) for the local tier, and every call that takes the retry-guard path performs another for the tier and one per pool. A compliance job that previously did one walk per erase now does three or more, saturating the vault's disk for the duration with no progress signal distinguishing a slow sweep from a hung one.
    → fix: Reduce the number of exhaustive walks rather than the reach of any one of them. In `eraseImpl`, thread the `erasureOutstanding` probe's result into the verdict instead of re-asking `holds` a second time on the same backend in `incompleteErasureFaults`. In `erasureOutstanding`, probe this gateway's backend and short-circuit before recursing into pools when there is nothing to sweep locally and no pool is attached. Keep the exhaustive fallback exactly as it is — a fast path that could answer ABSENT from a single stat would hollow the straggler and misfiled-copy rails — and record the measured cost of one whole-vault walk in the PR body so the bound is stated rather than assumed.
