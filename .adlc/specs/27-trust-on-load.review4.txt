
  [33mMEDIUM  [39m [35m[correctness][39m [1m`lawFrom` union correctness silently depends on an unstated manifest-retention invariant[22m  [2mconf 0.50[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    `lawFrom(M)` is specified as the authoritative incident-response answer to 'am I exposed through M?', computed as the union of currently-bound law across any manifest version of M the store holds. Its correctness therefore requires the store to still HOLD every manifest version a row was ever adopted from. The spec never states this as an invariant. If any historical manifest version is absent — never persisted, pruned, or garbage-collected — `lawFrom` omits rows adopted from it and reports 'not exposed' while the root still binds that law: precisely the miss round-3 review invoked to reject narrowing to the latest manifest. Loam's grow-only store likely satisfies retention by construction, but that load-bearing dependency is left implicit, so any future pruning or a store that did not retain the old manifest breaks the query silently. This is an inference about an absent invariant; confidence is held down accordingly.
    [33m✗ failure:[39m A root adopted a `Post` schema from `social@1`; the operator later upgrades to `social@7`, whose manifest no longer lists that row. During an incident the operator runs `lawFrom(social)`. If `social@1`'s manifest was not retained, the query returns the row as absent, the operator concludes they are not exposed, and the still-bound law keeps resolving root reads.
    [32m→ fix:[39m Name the retention invariant `lawFrom` depends on — the store retains every historical manifest version referenced by an adoption record (grounded in the grow-only store) — and add an acceptance criterion asserting `lawFrom` still reports a row after its source manifest version is superseded but retained.

  [33mMEDIUM  [39m [35m[other][39m [1m`blessAll` has no specified way to resolve a per-row root-name collision[22m  [2mconf 0.55[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    The collision resolutions `supersede` and `as <name>` are described only for a single adoption naming one target. `blessAll` is enumeration that, on a failing row, 'refuses the remainder loudly.' But a root-name collision is not an error the operator can fix by editing a stranger's module; it requires a per-row decision (take over vs. alias under a new name). The spec does not say how a bulk `blessAll` supplies those per-row decisions, whether a name collision aborts the remainder like a validation failure, or whether the operator must fall back to single adoptions. Criterion 6 covers a row that 'fails validation' but a guard refusal is a policy decision, not a validation failure — two implementers will build the vendor-suite collision UX differently.
    [33m✗ failure:[39m An operator runs `blessAll(vendor@3)` over a six-row suite; row 4's `Post` collides with an existing root `Post`. One build aborts after row 3 and forces the operator to re-run remaining rows singly with `as`; another treats the collision as a skippable row; a third refuses the whole call. The 'one gesture' promise and which rows land diverge across conforming builds.
    [32m→ fix:[39m Specify `blessAll`'s behavior when a row hits the root-name guard — either a mechanism to pass per-row `supersede`/`as` decisions into the bulk call, or an explicit rule that a guard collision aborts the remainder like any other refusal — and add an acceptance criterion for a `blessAll` whose enumeration contains a colliding row.

  [90mLOW     [39m [35m[concurrency][39m [1mGuard atomicity relies on a single-writer deployment with no stated precondition or runtime check[22m  [2mconf 0.50[22m
    [36m.adlc/specs/27-trust-on-load.md (file-level)[39m
    The guard's atomicity is implemented as 'an in-process queue' fronted by 'the single-writer gateway.' This closes the confused-deputy race only while exactly one writing gateway process fronts the store. The spec grounds on that doctrine but declares no invariant or rail that detects or fails when more than one writing gateway is deployed against the same store; an in-process queue cannot serialize across processes, silently reopening the race the guard exists to close. The spec invokes the doctrine but does not surface single-writer as an explicit precondition of the atomicity guarantee.
    [33m✗ failure:[39m An operator horizontally scales the gateway to two writer processes for availability. Two concurrent different-content adoptions of the same `Post` arrive on different processes; each in-process queue serializes only its own work, both name-checks pass against the same observed winner, and both publish — the exact double-publish criterion 14 asserts cannot happen.
    [32m→ fix:[39m State single-writer as an explicit precondition of the atomicity guarantee and describe what enforces or detects it (e.g. a store-level writer lease), or record multi-writer deployment as a known-unsafe configuration for this guard.
