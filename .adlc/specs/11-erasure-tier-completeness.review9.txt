Summary
  Do not ship as-is. The erasure-completeness rework is unusually well-railed at the gateway layer, but the new durable WAL-truncation debt — the mechanism that carries an incomplete §11 erasure across a process boundary — has two fail-open paths that reinstate exactly the hazard (H9) this branch just documented: its only automatic discharge point (the boot `heal` sweep) swallows its own failure with no reader anywhere in `src/`, and an unreadable debt row silently becomes 'nothing owed', after which `holds` answers false over plaintext that may still sit in the `-wal` sidecar. Both leave the operator with a store that reports a completion it never delivered.

  HIGH     [observability] The truncation debt's only automatic discharge point swallows its own failure: an incomplete §11 erasure survives a boot in total silence  conf 0.72
    src/cli/cli.ts:132-141
    This diff makes `SqliteBackend`'s WAL-truncation debt durable so a crashed partial erasure is still known after restart, and it deliberately un-gates the checkpoint so a purge that deletes nothing still attempts the truncation (`sqlite.ts:279`). At boot, the only thing that automatically drives that path is `MirrorBackend.heal(dead)`, which is handed the whole accumulated tombstone set from `cli.ts:132`. But `heal`'s `sweep` catches every purge refusal into `purgeFailures` (`mirror.ts:152-160`), and `cli.ts:137-141` reads only `toPrimary`/`toMirror`. A grep of `src/` confirms `purgeFailures` has no reader anywhere in the shipped product — the field's own comment claims the opposite (`mirror.ts:36`: "sweeps that refused — reported, never swallowed"), and this diff's new hazard entry admits it (H9, "Cost so far", filed as T70) while shipping it. The result is that the newly-reachable sqlite refusal — `"...so this erasure is INCOMPLETE (§11)"` — is discarded on the exact code path meant to settle it. `purgeFailures` predates this branch, but this branch is what makes it carry §11 completeness verdicts: before, `purge` only threw when `removed > 0`; now it also throws on a retry/boot sweep that matched nothing but owes truncation, which is precisely the post-crash boot shape.
    ✗ failure: An operator erases a record. The `wal_checkpoint(TRUNCATE)` is held by a concurrent reader, `purge` throws INCOMPLETE, and the debt row is persisted. The process is killed before the retry. On the next `loam serve`, `heal(dead)` calls `sqlite.purge(dead)`; the debt id is in `dead`, the checkpoint is still busy (a backup agent, a second handle), so `purge` throws again. `sweep` catches it into `purgeFailures`, `heal` returns normally, `cli.ts` prints nothing because `toPrimary`/`toMirror` are 0, and the store boots reporting health. The erased subject's plaintext remains legible in `store.db-wal`, indefinitely, with no signal to the operator and no automatic retry.
    → fix: Make `heal`'s `purgeFailures` load-bearing before this branch lands: have `cli.ts` write each entry to `io.err` (and exit non-zero, or refuse to serve, when a failure names a tombstoned id), and add a rail that boots a store with an owed, unsettleable debt and asserts the operator is told. Landing the durable debt without a reader for its failure ships the mechanism and drops its only alarm.

  MEDIUM   [data-loss] An unreadable truncation-debt row silently becomes "nothing owed", and the comment's stated recovery does not happen  conf 0.60
    src/store/sqlite.ts:104-116
    The constructor reads the persisted debt and, on any parse failure — or on a value that parses to a non-array, which is not even caught — leaves `truncationOwed` empty (`sqlite.ts:108-116`). The inline comment asserts "the purge path below re-earns or clears it", but it does neither for the case that matters. On the operator's retry, the rows are already deleted, so `deletedNow` is empty and the `if (deletedNow.length > 0)` block never re-adds the ids (`sqlite.ts:267`); `owedHere` is then false, so a busy checkpoint does not even throw (`sqlite.ts:286-287`); and any later purge whose checkpoint succeeds executes `DELETE FROM meta` (`sqlite.ts:284`), erasing the unreadable record entirely. Meanwhile `holds(id)` short-circuits on `truncationOwed` and, finding it empty and the row deleted, returns `false` (`sqlite.ts:326-330`) — a licence for `eraseImpl`'s verdict to report completeness. This is H9 verbatim ("could not determine" answered as "no") inside the file that this same diff added H9 to defend. It is a narrow trigger — a torn write, a truncated file, or a future change to the row's encoding — but the failure is silent, permanent, and lands on the one probe whose FALSE authorizes a compliance claim.
    ✗ failure: A crash or disk fault leaves `meta.value` truncated (e.g. `["abc`). The reopened handle's `JSON.parse` throws, the catch swallows it, and `truncationOwed` is empty. The operator re-runs `erase(id)`: `holds(id)` returns false, `eraseImpl` collects no faults, and `erase` resolves `{ erased: id }`. The pre-delete page images are still in `store.db-wal`, and the store has now told the operator — and the compliance record — that the bytes are gone.
    → fix: Fail closed instead of open: on an unparseable or non-array debt row, set a handle-level "debt of unknown extent" flag that makes `holds` return true for any id until a checkpoint succeeds (and never `DELETE FROM meta` while that flag stands), or refuse to open the store and tell the operator to run the truncation manually. Add a rail that writes a corrupt `truncation-outstanding` value, reopens, and asserts `holds` does not answer false.

  LOW      [error-handling] An absent `wal_checkpoint` result is treated as a successful truncation in both purge and close  conf 0.45
    src/store/sqlite.ts:343-347
    Both new checkpoint sites read the pragma result and treat `undefined` as success: `purge` computes `const busy = status !== undefined && status.busy !== 0` (`sqlite.ts:280`) and `close` clears the debt on `status === undefined || status.busy === 0` (`sqlite.ts:344`). In `close` this is the stronger claim — an empty pragma result causes both `truncationOwed.clear()` and `DELETE FROM meta`, discarding a durable obligation on the strength of a checkpoint that returned no answer at all. This is the same H9 shape as finding #2 and lives one screen below the code that added H9 to the hazards file. I could not construct a driver path where better-sqlite3 returns an empty array for `wal_checkpoint` on a WAL database, so this is an inference about a defensive branch the code itself chose to write; the severity reflects that. The `purge` form is inherited from `main`, but `close`'s debt-clearing use of it is new here.
    ✗ failure: Any condition under which `db.pragma("wal_checkpoint(TRUNCATE)")` yields no row — a journal-mode fallback on a network filesystem, a driver upgrade that changes the return shape — makes `close()` clear a real truncation debt and delete its persisted record. The next handle reports `holds === false` for an id whose pre-delete images are still in the sidecar, and a retry is refused as `nothing to erase`.
    → fix: Invert the default at both sites: treat a missing or unrecognized `wal_checkpoint` result as BUSY (debt stands, `holds` stays true) rather than as a completed truncation, and assert the shape you got rather than destructuring optimistically.
