 NEEDS ATTENTION   branch t67-tier-blind-erasure vs main (merge-base 060d7a2ec780b58fbd23a24b4d97e88ed86b7283)

Summary
  The core §11 fix is sound and its rails are largely non-hollow: `holds` is a genuine byte-presence question, `MirrorBackend.holds` probes both tiers and fails closed with the tier named, the pool fan-out is settled rather than raced, and the retry anchor is correctly narrowed to a SURVIVING tombstone bounded by outstanding work. What I would not ship without a second look is the new `truncationOutstanding` latch: it is the only record that a sqlite purge left WAL truncation owed, it lives in per-handle memory while the erasure it belongs to is durable, and `holds` — now promoted to THE completeness verdict — cannot see it. After an unclean restart the documented retry is refused as `nothing to erase`, so no invocation of `erase` can finish that truncation, which is the same stranding class the ticket was written to delete, merely moved across a process boundary. Three smaller defects ride along: the replica-side verdict still throws ahead of the nested refusals it collected (and its own comment claims otherwise), the guard's new `heldAnywhere` probe turns a repeat erase into a full vault walk on every tier and pool, and the citation filter drops surviving struck tombstones rather than only the erasure's own.

  MEDIUM   [data-loss] The outstanding-WAL-truncation obligation is per-handle memory and invisible to `holds`, so after a restart the documented retry is refused as `nothing to erase` and no erase can finish the truncation  conf 0.55
    src/store/sqlite.ts:231-246
    `SqliteBackend.purge` now records unfinished truncation in a private instance field (`private truncationOutstanding = false;`) and refuses only when `removed > 0 || this.truncationOutstanding`. That is the right shape for the in-process retry, and the new rail in `test/store/erasure-at-rest.test.ts` exercises it at the DRIVER. But the obligation is stored nowhere durable, while the erasure it belongs to (the tombstone) is append-only and survives forever, and `holds` — which this PR promotes to THE §11 verdict — is `SELECT 1 FROM deltas WHERE id = ?` and cannot see it.
    
    Trace the flow through `erase`, not through `purge`. Attempt 1: the tombstone lands, `gw.backend.purge([id])` (erase.ts:305) deletes and COMMITs the rows, the TRUNCATE checkpoint returns busy, the latch is set, and `purge` throws — so `gw.reseat()` at erase.ts:306 never runs and the target stays in the (now stale) reactor. That is the only reason the in-process retry works: the guard passes on `target !== undefined`, not on any knowledge that truncation is owed.
    
    Now the process ends without a graceful `close()` (crash, SIGKILL, container eviction). The new handle has `truncationOutstanding === false` and the reactor, re-seated from the store, no longer holds the target because its rows really were deleted. The operator does exactly what the error text instructs — retry — and `eraseImpl` evaluates `target === undefined && (already === undefined || !(await heldAnywhere(gw, id)))`: the tombstone survives so `already` is defined, but `heldAnywhere` asks `holds`, the row is gone, and the answer is false. The call rejects with `nothing to erase`. No path through `erase` can drive the checkpoint the first attempt left owed. For a plain sqlite store there is no boot sweep at all (`src/cli/cli.ts:132` calls `heal` only for a mirror), and for a mirror the sweep's failures land in `purgeFailures`, which this PR itself files as unread (T70).
    
    Honest bounds, stated because they matter to severity: a graceful `close()` checkpoints and clears the residue, and any later checkpoint (an auto-checkpoint from subsequent writes) folds the zeroed pages back over the plaintext. So the exposure is an unclean-termination window, not a permanent leak. What is unconditional is the stranding and the misleading rejection — the exact failure mode the working spec says the tombstone anchor was introduced to eliminate, reappearing because the anchor's companion signal (is anything still owed?) was not made equally durable. This is an inference from reading the post-change source; I did not run it.
    ✗ failure: An operator runs a single-file sqlite store. A backup agent holds a read transaction. `erase(X)` records the tombstone, DELETEs and COMMITs X's rows, then `wal_checkpoint(TRUNCATE)` returns `busy: 1`; the latch is set and `purge` throws, so `erase` rejects with 'Retry once the other handle is idle.' Before the operator retries, the host is evicted and the process dies without closing the database. On restart the latch is gone and X is not in the reactor. The operator retries `erase(X)` as instructed and is told `nothing to erase: X is not held here`. `holds(X)` is false, so every completeness probe in the system now reports the store clean, while X's pre-delete page images may still be legible in `store.db` until some unrelated future write triggers a checkpoint. The compliance record shows a rejected erasure that no re-run can ever clear.
    → fix: Make the obligation as durable as the erasure it belongs to, or make the retry guard consult it. Cheapest correct fix: persist the outstanding-truncation flag where the store can find it on reopen (a one-row `meta` table written in the same transaction as the DELETE, or a sentinel file beside `store.db`), and have `SqliteBackend.holds` return true — or expose a separate `truncationOutstanding` the gateway also treats as a fault — while it is set. Then `heldAnywhere` lets the retry through and the checkpoint is actually re-attempted. Add a rail that spans the boundary: fail the checkpoint, discard the handle WITHOUT calling `close()`, reopen, and assert (1) `erase(id)` does NOT reject with `nothing to erase`, and (2) after the retry the marker bytes are absent from every file in the directory. Until that exists, state in the spec that the retry criterion holds only within a single process lifetime, rather than letting the landed §11 prose imply otherwise.

  LOW      [observability] `eraseReplicaImpl` throws its local retention verdict before the nested refusals it collected, and its own comment claims the opposite  conf 0.80
    src/gateway/erase.ts:445-464
    The walk in `eraseReplicaImpl` is settled with `Promise.allSettled` and the first rejection is kept in `refused`, but the local byte-presence verdict is thrown first and `if (refused !== undefined) throw refused.reason;` runs only after it. So whenever a pool both retains its own bytes and has a nested replica that refused, the nested refusal is discarded and never reaches the operator. Worse, `await gw.backend.holds(id)` on the line above is NOT wrapped: if this pool's tier cannot be asked at all (an unreachable durable backend), that raw rejection propagates and every collected nested refusal is dropped with it.
    
    The comment immediately above the walk asserts the opposite property — that the walk is settled so 'a nested replica that cannot be proven clean must not hide the sibling that also could not' — and cites `MirrorBackend.purge`'s settle-then-report semantics as the model. `eraseImpl` genuinely does aggregate (`incompleteErasureFaults` collects the local fault and every rejected fan-out result into one message); `eraseReplicaImpl` does not, so the two halves of the same fix disagree. Only `refused.find` is kept in any case, so N sibling refusals surface one at a time even on the path that works.
    
    This never produces a false completion — the call still rejects — so it is a diagnosability defect on the compliance path rather than a correctness hole. It matters because the remedy the error prescribes is 'Resolve the pool's store fault and re-run', and the operator is handed one fault per round trip out of a set the code already had in hand.
    ✗ failure: An operator has pool P attached, with nested pools R1 and R2 beneath it; R1's backend silently retains and P's own backend also retains. `erase(X)` fans to P: P federates the tombstone, purges, walks R1 and R2 concurrently, and R1 rejects. `refused` holds R1's reason. P then finds it still holds X itself and throws its own 'STILL HOLDS' error; `refused` is discarded. The primary reports one fault naming P. The operator repairs P, re-runs, and only now learns about R1 — two round trips for a fault set that was fully known on the first call. If P's tier had been unreachable rather than retaining, R1's refusal would have been dropped by the unguarded `holds` rejection instead.
    → fix: Give `eraseReplicaImpl` the same shape `eraseImpl` already has: build a fault list rather than throwing at the first one — wrap `await gw.backend.holds(id)` in try/catch so a tier that cannot be asked becomes a fault instead of an escape hatch, collect every rejected entry of `walked` (not just `.find`), and throw one error enumerating each pool that could not be proven clean, carrying the first as `cause`. Factor it through the existing `incompleteErasureFaults` helper so the two paths cannot drift again. Then either fix the comment above the walk or make the code match it. Extend the nested-pool rail in `test/gateway/erasure-fanout.test.ts` to plant retention at BOTH a pool and one of its children and assert the message names both.

  LOW      [resource-exhaustion] The new retry guard turns a repeat `erase` of an already-erased id into a full byte-presence walk of every tier and every nested pool  conf 0.70
    src/gateway/erase.ts:348-360
    The `nothing to erase` guard is now `target === undefined && (already === undefined || !(await heldAnywhere(gw, id)))`. Whenever a surviving tombstone names the id — which is true forever for every id this store has ever erased — the short-circuit fails and `heldAnywhere` runs, probing `gw.backend.holds(id)` and then recursing into every attached quarantine pool and every pool nested beneath them. Under a mirror that is two tier probes; if either tier is an `ArchiveBackend`, `holds` lists every fan directory under the vault root and scans every filename in each (`readdirSync(this.root, ...)` then `readdirSync(join(this.root, fan))` per fan, with a per-name suffix test). `incompleteErasureFaults` then asks `holds` a second time on the paths that get past the guard.
    
    Before this change, `erase` on an id the reactor did not hold threw immediately with zero I/O. The repo's own H8 hazard is exactly this shape — a full walk introduced on a path whose cost nobody measured because 'erasures are rare'. Repeat erases are not rare in the same way: re-running a bulk subject request, replaying a compliance script, or a client that retries idempotently all hit this guard, and each call now pays a whole-vault walk per tier.
    
    This is a cost finding, not a correctness one, and the walk is genuinely the right reach for the verdict — the objection is to paying it on the guard, for ids that are provably gone, on a path that previously cost nothing.
    ✗ failure: An operator re-runs last quarter's subject-request script — 1,000 `erase(id)` calls, all of which completed months ago — against a mirror(sqlite, archive) store holding 100,000 archived deltas. Each call finds a surviving tombstone, so each calls `heldAnywhere`, so each does a sqlite row lookup plus a full archive walk: ~256 readdir syscalls and ~100,000 filename comparisons, twice over if a pool is attached. The run does roughly a quarter of a million directory reads and 10^8 string comparisons to produce 1,000 `nothing to erase` rejections, and the operator sees a compliance job that used to finish instantly now hammering the vault's disk.
    → fix: Order the guard so the cheap discriminator runs first: probe only this gateway's backend before recursing into pools, and short-circuit the pool walk when the local tier is clean AND no pool exists. Better, bound the probe to erasures that could plausibly be outstanding — e.g. only run `heldAnywhere` when the tombstone's own erasure has not been recorded as completed, or memoize per (id, reactor generation) so a repeated call in one session does not re-walk. If the walk stays unconditional, give `ArchiveBackend.holds` the fast path first (stat the two canonical names in the id's own fan) and fall back to the exhaustive cross-fan walk only when those miss — the exhaustive form is what the straggler and misfiled-copy rails need, and it still runs whenever the targeted lookup says absent.

  LOW      [correctness] The citation filter drops every tombstone naming the id, not just the erasure's own — so a surviving struck tombstone vanishes from the manifest it belongs in  conf 0.60
    src/gateway/erase.ts:283-288
    The manifest is documented as 'every delta citing the id … the holes the cut will leave', and the new filter's stated purpose is narrow: 'The erasure's OWN tombstone is not a citation.' The code is wider than the reason given — it removes any delta that is a tombstone whose `targetId` is the id, regardless of whether it is the tombstone being minted or reused.
    
    The divergent case is reachable and is exercised elsewhere in this very diff: erase X (tombstone T1), strike T1 as forgiveness, re-admit X, erase X again (tombstone T2, which the new surviving-tombstone anchor correctly mints fresh). T1 is struck but still in the ground, still carries an `erases` pointer at X, and still survives the second cut — it is precisely a hole the cut leaves behind, and a caller reading `citations` to understand the blast radius will not be told about it. The rationale for the filter (a caller cascading on citations would try to erase the cut itself and be refused by the append-only guard) does not apply to T1, which is not the cut being made.
    
    Low severity: the manifest is advisory and cascade is the caller's choice. It is worth fixing because the filter is broader than its own comment, which is how a narrow intent becomes a wrong behavior nobody can later reconstruct.
    ✗ failure: An operator erases X, later strikes the tombstone as forgiveness, and X is re-admitted from a peer. A second subject request arrives and the operator runs `erase(X)` again, then reads the returned `citations` to decide what else to cascade. The struck tombstone T1 — a surviving delta that points at the now-purged X and is exactly the kind of dangling reference the manifest exists to enumerate — is absent from the list, so the operator's cascade misses it and the audit of what the cut left behind is incomplete.
    → fix: Filter by identity rather than by shape: compute the tombstone (the `already ?? signClaims(...)` result) first and exclude exactly that delta id from the citation list, leaving any other tombstone that cites the id in place. If the tombstone must be computed after the manifest for ordering reasons, restrict the predicate to SURVIVING operator tombstones (reuse `survivingTombstones`, which the retry anchor already calls) so a struck one is retained. Extend the existing 'the manifest never cites the erasure's own tombstone' rail with the forgiveness case: after strike + re-admit + re-erase, the struck tombstone MUST appear in `citations` while the new one MUST NOT.
