Summary
  The core §11 fix is sound — `holds` is a real byte-presence question, the mirror probes both tiers and fails closed, and the new rails are non-hollow (the headline rail genuinely goes red with the verdict reverted). What should not ship as-is is the ride-along change to `SqliteBackend.purge`: dropping the `removed > 0` gate on `wal_checkpoint(TRUNCATE)` is correct for the documented retry, but it also fires on every purge that matched nothing — which is the common case for a quarantine pool that never held the id and for the boot `heal` sweep. Under a concurrent reader that now rejects, so a fully-completed erasure gets reported INCOMPLETE and pointed at a replica that never had the record, and on the boot path the same failure lands in `purgeFailures`, which the PR itself documents (T70) as having no reader. Two smaller defects: both new `Promise.allSettled` sites report only the first of N refusals, and the local verdict throws ahead of the pool refusal — directly contradicting the comment that claims neither masks the other. Fix the checkpoint gating (or bound it to outstanding truncation work) and re-run before landing.

  MEDIUM   [error-handling] Ungating the WAL checkpoint makes every purge that matched nothing able to fail the whole erasure — including on tiers and pools that never held the record  conf 0.65
    src/store/sqlite.ts:216-237
    `SqliteBackend.purge` now runs `wal_checkpoint(TRUNCATE)` unconditionally and converts a `busy` result into a thrown 'erasure is INCOMPLETE (§11)'. That is the right fix for the one path the comment describes (first attempt deleted the rows, failed to truncate, retry finds `removed === 0`). But the gate that was removed was also the only thing keeping the checkpoint off purges that had nothing to do with an outstanding truncation, and this diff simultaneously multiplies how often those happen:
    
    1. `eraseReplicaImpl` calls `await gw.backend.purge([id])` on EVERY attached quarantine pool (erase.ts:358). `QuarantineOptions.backend` is documented as taking a durable store ('pass a durable backend for a long-running quarantine', quarantine-pool.ts:37-39). A sqlite-backed pool that never held `id` now forces a TRUNCATE checkpoint on its own database; under any second handle holding a read it blocks for `busy_timeout` (5000ms, sqlite.ts:56) and then rejects. That rejection propagates out of `Promise.allSettled` and fails the primary's `erase`, which had already purged the record from every tier that actually held it.
    2. `MirrorBackend.heal` sweeps with the whole accumulated tombstone set on every `loam serve` boot (`healed = await mirror.heal(dead)`, cli.ts:132). Those DELETEs match nothing on a healthy store, so before this change no checkpoint ran; now one runs on every boot. When it rejects, `sweep` catches it into `purgeFailures` (mirror.ts:157-159) — and `cli.ts:137-141` reads only `toPrimary`/`toMirror`, which is exactly the unread channel this PR files as T70. The boot prints its normal line and serves.
    
    So the change trades a real false-completion for (a) an availability regression on the compliance path that reports a COMPLETE erasure as incomplete and directs the operator at a replica that never held the record, and (b) a higher firing rate into a failure channel the PR itself documents as having no reader. Inference note: I did not execute this; the conclusion rests on the code as it now stands plus the driver's own comment that the pragma 'gives up with busy: 1' rather than throwing.
    ✗ failure: An operator runs a long-running quarantine pool on `SqliteBackend`, seeded with a `membership` Term that excludes record X. A `loam repair` session (or a backup agent) holds a read transaction on the pool's database. The operator runs `erase(X)`. The tombstone lands, the primary and mirror purge X's bytes, and the fan-out reaches the pool: its DELETE matches nothing (`removed === 0`), the now-unconditional `wal_checkpoint(TRUNCATE)` waits out `busy_timeout` and returns `busy: 1`, and `purge` throws 'this erasure is INCOMPLETE (§11)'. The pool's `eraseReplica` rejects, so `erase(X)` rejects. The compliance record shows a failed erasure for a record that is in fact gone everywhere, the error names a pool that never held it, and every re-run fails identically for as long as the unrelated reader stays open.
    → fix: Keep the checkpoint unconditional but bound the FAILURE to outstanding truncation work rather than to every call: latch a per-handle flag when a post-DELETE checkpoint is refused (or when rows were deleted in this transaction), and convert `busy` into a rejection only when `removed > 0 || truncationOutstanding`. That covers the retry path the comment describes without letting a purge for an id this store never held fail an erasure that completed. Add two rails: (1) `purge([id])` for an id the store never held, with a concurrent reader holding the WAL, must NOT reject; (2) the existing retry rail in erasure-at-rest.test.ts stays red if the latch is dropped. Separately, land T70's `serve`-side reader for `purgeFailures` before widening how often that channel fires — a boot-path failure with no reader is the same H7 shape this ticket exists to delete.

  LOW      [observability] Both new settle-then-report sites surface only the first refusal, and the local verdict throws ahead of the pool refusal — contradicting the comment that claims neither masks the other  conf 0.85
    src/gateway/erase.ts:305-319
    `eraseImpl` and `eraseReplicaImpl` both compute `fanned`/`walked` with `Promise.allSettled`, then take `.find((r) => r.status === "rejected")` — a single result. If three replicas each refuse, the operator learns about one, repairs it, re-runs, and learns about the next. The two `holds` verdicts have the same shape one level up: `MirrorBackend.holds` throws on the FIRST rejected tier, so a store with two unreadable tiers names one.
    
    Worse, the comment at erase.ts:317-318 asserts an ordering the code does not implement: 'reported last so the local verdict is never masked by a remote one, nor the remote by the local.' The local `if (await gw.backend.holds(id)) throw` runs BEFORE `if (poolRefused !== undefined) throw poolRefused.reason`, so whenever the local tier also retains — or its `holds` rejects, which is the unreachable-tier case this PR deliberately made rejecting — the replica refusal is discarded entirely and never reaches the operator. `eraseReplicaImpl` repeats the same ordering at lines 384-391.
    
    This never produces a false completion (the call still rejects), so it is a diagnosability defect on the compliance path rather than a correctness hole. It matters because the remedy the error text prescribes is 'resolve the store fault and re-run', and the operator is being handed one fault at a time out of a set the code already collected.
    ✗ failure: A store has an unreachable archive mirror and two attached quarantine pools whose backends both silently retain. The operator runs `erase(X)`. The fan-out settles and collects two rejections; `poolRefused` keeps the first. Then `gw.backend.holds(id)` rejects because the mirror tier cannot be proven clean, and that error propagates — both pool refusals are dropped on the floor. The operator remounts the archive and re-runs, is told about pool 1, fixes it, re-runs, is told about pool 2. Three round trips and three partial reports for a fault set that was fully known on the first call.
    → fix: Collect every rejection rather than the first, and report the local and remote verdicts together: build the local `holds` result and the settled fan-out results into one `AggregateError` (or a single message enumerating each tier/pool that could not be proven clean, each with its `cause`), thrown after both are known. Apply the same shape in `eraseReplicaImpl` (erase.ts:378-391) and in `MirrorBackend.holds` (mirror.ts:89-97), which already has both tiers' results in hand. Then either fix the comment at erase.ts:317-318 or make the code match it — as written it documents behavior the function does not have.

  LOW      [correctness] Moving the retry anchor ahead of the guard removes the only check that `erase` did any work, so it now resolves `{ erased }` on a store that holds nothing and purged nothing  conf 0.80
    src/gateway/erase.ts:250-260
    The reorder is necessary — after a partial attempt the target is routinely gone from the reactor, and gating the re-run on `gw.reactor.get(id)` would strand the sweep. But the new predicate is `target === undefined && already === undefined`, so ANY surviving operator tombstone naming `id` suppresses the guard, not just one from an outstanding attempt. Once past it, `already !== undefined` also skips the append, `purge` removes nothing, `holds` is false, and the function returns `{ erased: id, citations }`.
    
    Concretely: `erase(X)` on an id erased successfully months ago now resolves instead of throwing `nothing to erase`. The same holds inside a quarantine pool, which receives the operator's tombstone unconditionally through the seeding edge (`admit: (d) => isTombstone(d.claims) || admit(d)`, quarantine-pool.ts:101) whether or not it ever held the record — `pool.gateway.erase(X)` reports a completion for a record that was never there.
    
    The report is not false in the §11 sense (the bytes genuinely are absent), which is why this is low rather than the H7 shape the ticket targets. It is still an unannounced change to what a public API returns, on the one call operators script against, and it is unrailed: the only new rail for this branch is `a struck tombstone is not a licence to report a completion`, which covers the case that still REJECTS. Nothing pins the case that newly resolves. A second-order effect on the same path: `citations` is computed from `gw.reactor.snapshot()` filtered by 'any delta with a delta-pointer at `id`', and on any call where the tombstone is already in the ground the tombstone matches its own `erases` pointer — so the blast-radius manifest returned by a retry includes the erasure's own tombstone, which a caller cascading on citations would then try to erase and be refused by the append-only guard.
    ✗ failure: An operator scripts a bulk subject request as 'call erase(id) for each id; a rejection means this store never held the record, so log it as out of scope.' After T67, every id the store erased in a previous request — and every id whose tombstone merely fanned into an attached pool — returns `{ erased: id }` instead of rejecting. The audit trail the script produces claims this run erased records it did not touch, and the 'never held here' column is silently empty.
    → fix: Narrow the bypass to an erasure that is actually outstanding rather than to any surviving tombstone: keep the anchor, but only skip the `nothing to erase` guard when the store still has something to sweep — e.g. proceed when `target !== undefined || await gw.backend.holds(id)`, and otherwise reject as before. That preserves the headline retry case (b) (the mirror still holds the bytes, so `holds` is true even though the reactor lost the target) while restoring the guard for an id that is genuinely gone. Add the missing rail in `test/gateway/erase-tier-completeness.test.ts` alongside the struck-tombstone one: a second `erase` of a cleanly-erased id must reject. Separately, exclude the tombstone itself from `citations` (filter out deltas whose `erases` pointer is the one being minted/reused) so the manifest does not vary between the first call and the retry.
