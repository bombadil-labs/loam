Summary
  Do not ship as-is. The fix to the §11 verdict itself looks sound and its rails are non-hollow, but three things ride along that shouldn't. (1) The same PR rewrites its own CI rail-freeze gate from file-existence to base-declaration — and because every ticket on `main` carries `"rails": []` and CLAUDE.md deletes a ticket at landing, the rewritten gate freezes nothing, now or after this lands; it is also precisely what lets this branch edit three existing rail files that the old gate would have failed. (2) The retry anchor `already` was widened to bypass the `nothing to erase` guard but still matches STRUCK (forgiven) tombstones, which `survivingTombstones` deliberately excludes — so `erase` can reuse a forgiven tombstone, purge the bytes, and leave no surviving tombstone in the dead set, or report `{erased}` having erased nothing. (3) The new pool-side `holds` verdict sits before the transitive fan-out and inside a sequential loop, so a retaining pool now aborts tombstone delivery to sibling and nested pools that previously received it.

  HIGH     [data-loss] The widened retry anchor matches STRUCK (forgiven) tombstones, so erase can purge bytes while leaving no surviving tombstone — or report success having erased nothing  conf 0.70
    src/gateway/erase.ts:246-254
    `already` is now read before the existence guard and, when found, suppresses both the `nothing to erase` throw and the minting of a new tombstone. But its predicate is only `isTombstone && author === operator && targetId === id`. It does not consult lawful negation, whereas `survivingTombstones` (erase.ts, used by `readTombstones` and `forgottenSince`) explicitly skips `negated(delta.id)` — 'struck = forgiven'. Forgiveness is a shipped, tested feature (`test/gateway/erase.test.ts:137`, 'forgiveness is striking the tombstone: the id may then return').
    
    Two consequences, one of them new to this diff:
    
    (a) After forgiveness and re-admission, a second `erase(X)` finds the STRUCK tombstone, takes the `already` branch, appends nothing, and purges X. `readTombstones` still excludes X because the tombstone is struck — so the store has destroyed the bytes while its own dead set says X was forgiven, admission will re-admit X, and `forgottenSince` reports no erasure. This path existed before the diff (X was live, so the old guard passed) but the diff documents it as 'a SURVIVING lawful operator tombstone' and makes the tombstone the authoritative anchor, which cements it.
    
    (b) NEW: with the guard bypassed, `erase(X)` for an id the store does not hold now returns `{ erased: X }` instead of throwing, whenever any operator tombstone (including a struck one) names X. That is a completion report for work that was never done — the exact H7 shape the ticket exists to delete — and there is no test anywhere in `test/` asserting the `nothing to erase` behavior (grep for the string returns nothing outside `src/`).
    
    The two halves of the change also disagree with each other: `eraseReplicaImpl` verifies its landing with `readTombstones(...).has(id)`, which honors strikes, while `eraseImpl`'s anchor does not.
    ✗ failure: Operator erases X (tombstone T). Later they strike T as forgiveness and X is re-admitted from a peer. A second subject request arrives and the operator runs `erase(X)` again. `already` matches the struck T, no new tombstone is appended, X's bytes are purged on every tier, `holds` is false, and `erase` resolves `{ erased: X }`. The store now has no surviving tombstone for X: admission will accept X's return from any federated peer, `forgottenSince` will not confess the erasure, and the compliance record shows one erasure that was subsequently forgiven — while the data is in fact gone and its removal unrecorded.
    → fix: Reuse the reader that already encodes the rule: select the anchor from `survivingTombstones(gw.reactor, gw.operatorAuthor)` filtered by `tombstoneParts(...).targetId === id`, so a struck tombstone is not an anchor and a fresh one is minted. Keep the `nothing to erase` guard when neither a live target nor a SURVIVING tombstone exists, so a second erase of an already-erased id still cannot report a completion it did not perform. Add rails for both: (1) erase after forgiveness mints a second tombstone and the id is in `readTombstones` afterward; (2) `erase(id)` with no target and only a struck tombstone rejects.

  MEDIUM   [correctness] The new pool verdict runs before the transitive fan-out, so a retaining pool now blocks tombstone delivery to sibling and nested pools that previously received it  conf 0.70
    src/gateway/erase.ts:341-360
    In `eraseReplicaImpl` the `holds` check is placed after `purge`/`reseat` but BEFORE `seen.add(gw)` and the loop over `gw.quarantinePools`. In `eraseImpl` the fan-out is `for (const pool of gw.quarantinePools) await pool.eraseReplica(...)` — sequential, so the first rejection aborts the loop.
    
    Before this change a silently-retaining pool returned a purge count that was discarded, so the walk completed: every sibling pool and every nested pool still received the operator's tombstone (the thing that makes a replica refuse the id's re-entry) and still got purged. After the change, the first retaining pool throws, and pools ordered after it — plus every pool nested under the retaining one — receive neither the tombstone nor a purge. The retry re-runs from the start with a fresh `seen`, hits the same pool, and fails identically, so those replicas stay unswept and un-tombstoned for as long as the faulty pool is broken.
    
    The change is right to be loud, but it trades a silent leak in one replica for a blocking leak across all replicas ordered behind it. The repo already has the correct pattern for this: `MirrorBackend.purge` and `MirrorBackend.close` use `Promise.allSettled` and explicitly 'attempt both, then report the first refusal'; the new `MirrorBackend.holds` adopts it too. The fan-out does not.
    
    The new rails do not cover it: `erasure-fanout.test.ts` plants retention in exactly one pool, and the nested case has the retaining backend at the DEEPEST level, where nothing is ordered behind it.
    ✗ failure: An operator has two quarantine pools attached, P1 (a read-only mount that silently retains) and P2 (healthy, with a nested pool P3). `erase(X)` records the tombstone, sweeps the primary, then calls `P1.eraseReplica`, which purges nothing, sees `holds(X) === true`, and throws. P2 and P3 are never visited: they keep X's bytes AND have no tombstone for X, so they will re-admit and can re-federate X back into the primary once the operator clears the local failure. Before this change P2 and P3 were swept and tombstoned on the same call.
    → fix: Move the `holds` verdict to AFTER the transitive fan-out in `eraseReplicaImpl` (mirroring `eraseImpl`, where the verdict is deliberately last, 'after re-seating and after the pool fan-out'), and settle the whole walk before reporting: run the pool loop with `Promise.allSettled` (or collect failures and rethrow an aggregate naming every pool that could not be proven clean), exactly as `MirrorBackend.purge`/`close` already do. Add a rail with a retaining pool ordered BEFORE a healthy pool, asserting the healthy pool still received the tombstone and lost the bytes.

  LOW      [observability] Ungating the WAL checkpoint makes the failure message assert a deletion that did not happen, and forces a TRUNCATE checkpoint on every heal sweep  conf 0.55
    src/store/sqlite.ts:216-230
    Dropping `if (removed > 0)` is the right fix for the documented retry path, but the error text was written for that path and is now emitted unconditionally: 'The rows are deleted, but their plaintext may remain in the -wal sidecar' is false when `removed === 0` because the id was never in this table. An operator reading it will believe a partial erasure occurred on a store that never held the record.
    
    Second-order: `MirrorBackend.heal` calls `tier.purge(ids)` with the whole accumulated tombstone set on every boot (it short-circuits only when the set is empty). Previously a boot where nothing needed removing skipped the checkpoint; now every boot with any historical tombstone forces `wal_checkpoint(TRUNCATE)` on the sqlite tier, which can return `busy: 1` under any concurrent reader. `heal` catches it into `purgeFailures` so this is not an outage, but it converts a quiet no-op into a recurring reported failure with a message that misdescribes what happened.
    ✗ failure: A store with 200 historical tombstones boots while a backup agent holds a read transaction. `heal` sweeps the primary sqlite tier, the DELETEs match nothing (`removed === 0`), the now-unconditional TRUNCATE checkpoint returns `busy: 1`, and the boot report carries 'The rows are deleted, but their plaintext may remain in the -wal sidecar, so this erasure is INCOMPLETE (§11)' for records that were purged and checkpointed months ago. The operator opens an incident over a false partial-erasure report.
    → fix: Branch the message on `removed`: when `removed > 0` keep the current wording; when `removed === 0` say that no rows matched and the outstanding work was the truncation, which could not be completed. Keep the checkpoint unconditional.
