Summary
  The tier-blindness fix itself is well-constructed — `holds` is a real byte-presence probe, the fan-out settles instead of racing, and the rails assert at the tier rather than through `get(id)`. Do not ship as-is: two of the new fail-closed guarantees do not hold. `reseat()` still throws after its destructive teardown (the catch-up loop), leaving the gateway serving the erased record from a stale in-memory reactor after the bytes are gone; and the persisted WAL-truncation debt fails OPEN on a debt row that parses to anything other than an array, which is the exact H9 direction the code claims to close. Both are narrow-path but both defeat §11 promises the diff was written to make.

  MEDIUM   [correctness] `reseat()` can still throw after the destructive teardown, leaving the gateway serving the erased record from a stale reactor  conf 0.65
    src/gateway/gateway.ts:643-665
    The reordering comment states the whole point of reading first: "everything after it is destructive... Ordered the other way, a failed read left the gateway half-torn-down." That guarantee is only honored for the FIRST read (line 626). The catch-up loop at 652-664 runs AFTER the teardown (channels ended 635, sinks/lazyMats cleared 636-638, publicOpen dropped 639, resolverMemo cleared 642) and BEFORE the swap (`this._reactor = reactor`, 665). It has three throw paths: the 10-round quiesce cap (657), an ingest rejection (661), and an unhandled rejection from `this.backend.deltasSince(known)` (654) — the same backend whose purge may have just refused. On any of them the gateway is left in exactly the state the reordering was written to prevent: `_reactor`, `ingestVia`, persistence and `rebindImpl` all untouched, so the OLD pre-purge reactor keeps serving reads while every live subscription has already been terminated.
    
    Because `eraseImpl` runs `backend.purge([id])` BEFORE `reseat()`, this is a §11 breach and not merely a broken re-seat: the bytes are gone from the store, `holds` answers false, and the only fault reported is the reseat error — while GraphQL reads and materializations continue to serve the erased delta out of the stale reactor until the process restarts or a later `erase` retry manages a clean reseat. The 10-round cap is realistic on an animated gateway: the runner still watches the OLD reactor during reseat (it is only detached by the swap), so a firing binding emitting derived deltas can keep the differential read non-empty for ten consecutive rounds.
    
    This is inference from reading the control flow rather than an observed failure; I did not execute a hot-writer reseat.
    ✗ failure: An animated store with a binding runner emitting derived deltas is asked to erase a subject record. `backend.purge` succeeds. `reseat()` reads the store, tears down subscriptions, then enters the catch-up loop; the runner keeps appending, ten differential reads in a row come back non-empty, and line 657 throws. `eraseImpl` collects it into `localPurge`, `holds(id)` is false, so `erase` rejects with 'this store's purge refused: reseat: the store will not quiesce'. The operator sees a failed erasure — but the gateway is now serving reads from the pre-erase reactor, which still contains the record whose bytes were just purged, and every parked subscriber was killed. Any read of that entity returns the forgotten content.
    → fix: Make the catch-up loop non-destructive with respect to the gateway's live state: either (a) wrap lines 643-664 so that any throw still performs the swap onto the best reactor built so far plus `attachPersistence`/`rebindImpl` before rethrowing, or (b) move the whole read-and-quiesce sequence ahead of the teardown (read, loop to quiescence, only then end channels and swap — the remaining window is the teardown's own awaits, which the loop can be re-run over once). Add a rail that forces a quiesce failure (a backend whose `deltasSince` keeps returning new deltas, or one that rejects on the second call) and asserts the gateway does NOT still resolve the erased id afterwards.

  MEDIUM   [data-loss] Persisted truncation debt fails OPEN when the debt row parses to a non-array, silently discharging an erasure obligation  conf 0.60
    src/store/sqlite.ts:112-125
    The debt-recovery block claims a fail-closed reading: the `catch` sets `truncationUnknown = true` and the inline comment on line 118 says "a debt row that is not an id list still owes". But that `else` binds to the inner `if (typeof i === "string")` — it only fires for a non-string ELEMENT of an array. `if (Array.isArray(ids))` at line 115 has no `else` branch at all, so any `owed.value` that is syntactically valid JSON but not an array (an object, a number, a bare string, `null`) leaves `truncationOwed` empty AND `truncationUnknown` false. The handle then believes nothing is owed.
    
    That is the precise H9 inversion the surrounding code exists to prevent: `holds` (line 325) returns false for ids whose pre-delete page images may still be legible in the `-wal` sidecar, `incompleteErasureFaults` records no fault, and `eraseImpl` resolves `{ erased: id }`. An erasure reports completion over plaintext that is still at rest.
    
    Today the only writer emits `JSON.stringify([...set])`, so this is not reachable from this version's own writes — it is reachable from any future or concurrent version that changes the row's shape (e.g. to `{"v":2,"ids":[...]}`), and from external tampering with the meta table. Version skew is the realistic path, and it is silent: the old handle does not refuse, it forgets. I have not observed a malformed row in practice, which is why confidence is not higher; the code defect itself is unambiguous.
    ✗ failure: A later release changes the debt row format to an object to carry a schema version. An operator rolls back to this build (or a mixed-version fleet has both handles pointing at the same store file). The old handle opens the store, reads `{"v":2,"ids":["abc…"]}`, takes neither branch, and starts with zero debt and `truncationUnknown === false`. The prior purge's WAL truncation never landed, so the erased delta's plaintext is still in `store.db-wal`. `erase(abc…)` is retried: `holds` returns false, no fault is collected, and `erase` resolves `{ erased }` — a completion report for a record whose bytes a `strings store.db-wal` still yields.
    → fix: Add the missing `else` on the `Array.isArray` test: any present-but-unparseable debt row must set `truncationUnknown = true`. Restructure as `let ids: unknown; try { ids = JSON.parse(owed.value) } catch { ids = undefined }; if (Array.isArray(ids) && ids.every(i => typeof i === 'string')) { for (const i of ids) this.truncationOwed.add(i) } else { this.truncationUnknown = true }` — so the ONLY path that yields 'no debt' is a fully-validated id list. Cover it with a rail that writes a non-array value into `meta` directly and asserts `holds` answers true for an arbitrary id.

  LOW      [correctness] A busy checkpoint at `close()` leaves a debt row that later goes stale, turning `holds` into a permanent false positive and `erase` into a completion report for no work  conf 0.50
    src/store/sqlite.ts:332-352
    `close()` (332-352) only clears the persisted debt when its explicit `wal_checkpoint(TRUNCATE)` returns `busy === 0`, which the comment defends as "the honest state of a sidecar still unfolded". It is honest at that instant, but it does not stay honest: when the other handle holding the WAL later closes, SQLite checkpoints and unlinks the sidecar, and the persisted debt row is now stale with no mechanism to notice. On reopen, `holds(id)` returns true (line 325) for bytes that are provably gone.
    
    That phantom feeds directly into the new retry bypass in `eraseImpl`: `erasureOutstanding` sees `holds` true, the `nothing to erase` guard is skipped, and `erase` reuses the old tombstone, purges zero rows, clears the debt, and resolves `{ erased: id, citations }` — the 'completion report for no work' the `close()` comment itself names as the thing to avoid. The self-healing is real but it arrives one misleading success report late, and for a non-mirror store there is no boot sweep to clear it earlier (`heal(exclude)` is only wired for `MirrorBackend` in src/cli/cli.ts:132). The narrower variant is a spurious refusal: if the retry's own checkpoint is also busy, `owedHere` is true via `batch.some(id => this.truncationOwed.has(id))` and a completed erasure is reported INCOMPLETE.
    
    The test suite covers the graceful-close case ('a GRACEFUL close settles the debt it discharges: no phantom on reopen') but not the busy-close-then-sidecar-vanishes case.
    ✗ failure: A backup process holds a read transaction on store.db. An erasure's purge cannot truncate the WAL, so the debt for id X is committed and `erase` correctly refuses. The service is stopped; `close()`'s checkpoint is still busy, so the debt row persists. The backup finishes and its handle closes, and SQLite truncates and unlinks store.db-wal — X's plaintext is now genuinely gone. The service restarts, and the compliance operator re-runs `erase(X)` to confirm. Instead of the correct 'nothing to erase: X is not held here', the phantom debt drives the bypass and the call resolves `{ erased: X }` — a fresh completion record entered against an erasure that this call did not perform.
    → fix: Make the debt self-validating rather than purely durable: at open, if a debt row exists, check whether the `-wal` sidecar is absent or zero-length (or run a non-destructive `wal_checkpoint(TRUNCATE)` once) and clear the debt when the sidecar is provably empty, before any `holds` can answer from it. Failing that, have the bypass distinguish 'outstanding because bytes are held' from 'outstanding because unprovable' so a retry that finds nothing to purge resolves as `nothing to erase` rather than `{ erased }`. Add a rail: purge under a busy WAL, close while still busy, release the other handle, reopen, and assert `holds` is false and `erase` reports nothing to erase.
