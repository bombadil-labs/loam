Summary
  The core §11 fix is sound — `holds` is a real byte-presence question, `MirrorBackend.holds` probes both tiers and fails closed with the tier named, the pool fan-out is settled rather than raced, and the retry anchor is bounded by outstanding work. Two things should not land as-is. First, this branch commits five stray `null.<pid>.tmp` artifacts to the repository root, each a full signed delta — one of them the §11 erasure canary, another an operator tombstone — permanently in git history, in a PR whose entire subject is that byte copies of erased content must not survive; nothing gitignores the shape, so the next mutation run adds more. Second, the durable `truncationOutstanding` latch changes `purge` from 'fails when this call left truncation owed' to 'fails on EVERY call while the checkpoint is busy, for any id, forever across restarts' — and because the local purge sits unguarded ahead of the pool fan-out (erase.ts:306 before erase.ts:319), that throw denies the tombstone and the sweep to every attached quarantine pool, which is precisely the 'one replica's fault must not become every other replica's leak' rule the diff's own comments state. Clean the artifacts, bound the latched failure to the erasure that owns it, and re-run.

  MEDIUM   [error-handling] Once `truncationOutstanding` latches, every purge on that handle fails while the WAL is busy — including for unrelated ids — and the unguarded local purge then denies the tombstone and the sweep to every attached pool  conf 0.60
    src/store/sqlite.ts:246-270
    `SqliteBackend.purge` now attempts `wal_checkpoint(TRUNCATE)` unconditionally and refuses when `removed > 0 || this.truncationOutstanding` (src/store/sqlite.ts:246-252). The `removed > 0` half is correctly scoped to the call that owns the debt. The latch half is not scoped at all: while a concurrent reader keeps the checkpoint busy, the condition is true on EVERY subsequent `purge`, whatever ids are passed — including calls for ids this store never held. The latch is also now persisted to the `meta` table and re-read at open, so the condition survives process restart and applies to erasures months later that have nothing to do with the original truncation.
    
    The consequence is not confined to sqlite. In `eraseImpl` the local purge is `await gw.backend.purge([id]);` (src/gateway/erase.ts:306), unguarded, and it runs BEFORE the pool fan-out at erase.ts:318-321. A throw there aborts the function: `gw.reseat()` does not run, no pool receives the operator's tombstone via `eraseReplica`, and no pool is purged. The diff argues at length that this exact shape is unacceptable for a replica — 'One replica's fault must not become every other replica's leak' — and fixes it for the pool loop with `Promise.allSettled`, while leaving the local tier able to do the same thing to every pool at once. The new latch makes that abort reachable from an erasure the operator is not even performing.
    
    Second-order: `MirrorBackend.heal` sweeps with the whole accumulated tombstone set on every boot. On a latched sqlite tier with a busy checkpoint, that sweep now throws every time and lands in `purgeFailures`, which this PR itself files as having no reader (T70).
    
    Inference note: I read the post-change source and traced the call order; I did not execute the suite. The new rail `a purge that matched NOTHING does not fail on a busy checkpoint` covers only the unlatched case — it never sets the latch first, so the amplified path is unrailed.
    ✗ failure: A store erases record A while a backup agent holds a read transaction. The checkpoint returns `busy: 1`, `removed > 0`, so the latch is set in memory and written to `meta`, and `purge` throws. The host is restarted; the reopened handle reads the latch back as true. A new subject request arrives for record B, and the backup agent is again mid-run. `erase(B)` appends B's tombstone, calls `purge([B])`: the DELETE removes B's row, the checkpoint is busy, `this.truncationOutstanding` is true, and `purge` throws. `erase` rejects before `gw.reseat()` and before the fan-out, so the two attached quarantine pools receive neither B's tombstone nor B's purge — they keep B's bytes and remain able to re-admit B into the primary. Every retry fails identically for as long as the unrelated backup reader stays open, and the failure text points the operator at a WAL truncation owed by a different erasure.
    → fix: Bind the latched failure to the ids it actually covers rather than to the handle: record which ids had rows deleted under an untruncated WAL (a `meta` row carrying the outstanding id set, or a single flag consulted only when the purge's id set intersects it), so a purge for unrelated ids on a busy checkpoint does not fail. Independently, stop letting the local tier abort the fan-out: settle the local `purge` alongside the pool walk and feed its rejection into `incompleteErasureFaults` as one more fault, so a tier that refuses does not cost every replica its tombstone. Rail both: (1) latch the debt, then `purge` an unrelated id with the checkpoint busy and assert it does not reject; (2) make the local `purge` throw and assert every attached pool still received the tombstone and lost the bytes.

  LOW      [resource-exhaustion] `ArchiveBackend.holds` walks the entire vault — every fan directory and every filename — and the erase path asks it once per tier plus once per pool, twice on a retry  conf 0.70
    src/store/archive.ts:228-262
    `ArchiveBackend.holds` (src/store/archive.ts:228) lists every fan under the vault root and then reads every filename in each fan, running a suffix test and a string comparison per name. It cannot short-circuit on the id's own fan, by design — the misfiled-copy rail at `test/store/archive.test.ts` requires the cross-fan sweep — so the cost is O(all files in the vault) per call.
    
    `incompleteErasureFaults` calls `gw.backend.holds(id)` on every erase, and `eraseReplicaImpl` calls it again for every attached pool and every nested pool. On the retry path the guard's `heldAnywhere` adds another full walk per tier and per pool before any of that. Under `MirrorBackend(sqlite, archive)` that is at minimum one whole-vault walk per erase, and two per erase per archive-backed tier on any re-run.
    
    This is the repo's own H8 shape — 'Does this walk every delta, and is the affordance that avoids it correct?' — introduced on the compliance path. It is a cost finding, not a correctness one: the exhaustive reach is genuinely what the verdict needs, and erasures are rare. What makes it worth recording is that repeat and bulk erasures are not rare in the same way, and nothing in the diff measures or bounds it.
    ✗ failure: An operator replays a bulk subject-request script — a few thousand `erase(id)` calls — against a mirror whose cold tier is an `ArchiveBackend` holding ~100,000 archived deltas. Each call performs at least one full vault walk (~256 readdir syscalls plus ~100,000 filename comparisons), and each id that resolves through the retry guard performs two. The compliance job that previously finished in seconds now saturates the vault's disk for the duration, and an operator watching it has no way to tell a slow sweep from a hung one.
    → fix: Give `ArchiveBackend.holds` a targeted fast path before the exhaustive sweep: stat `<root>/<id[0..2)>/<id>.json` first and return true immediately on a hit, falling back to the cross-fan walk only when the canonical name misses (the fallback must stay, or the straggler and misfiled-copy rails go hollow). In `eraseImpl`, avoid asking the same backend twice on the retry path by threading the `heldAnywhere` result into the verdict rather than re-probing. Record the measured cost of a whole-vault walk in the PR body so the bound is not silent.

  LOW      [concurrency] The now-concurrent fan-out shares one mutable `seen` set, so the dedup guard is evaluated before any sibling has recorded itself  conf 0.40
    src/gateway/erase.ts:318-321
    `eraseImpl` builds `const seen = new Set<Gateway>([gw]);` (src/gateway/erase.ts:318) and then dispatches every pool at once through `Promise.allSettled` (erase.ts:319-321) — the top-level map does not consult `seen` at all. Inside `eraseReplicaImpl`, `seen.add(gw)` happens only after `federate`, `flush`, `purge` and `reseat` have awaited, and the child filter `.filter((pool) => !seen.has(pool))` (erase.ts:451-455) reads the set at that later moment. Under the previous sequential `for … await` the set was always fully up to date when read; under `Promise.allSettled` two concurrent branches can both pass the filter for the same gateway before either records it.
    
    `Gateway.quarantinePools` is a public mutable `Set<Gateway>` (src/gateway/gateway.ts:583), so a caller can attach the same pool gateway beneath two parents, which is what makes a diamond reachable. The comment's justification — 'a cycle among pools cannot arise from openQuarantine (each pool is a fresh gateway)' — is about `openQuarantine`, not about the set's public mutability, and it was written for the sequential walk.
    
    The blast radius is bounded: appending the same content-addressed tombstone twice is idempotent, and `purge` is idempotent, so the visible risk is two concurrent `federate`/`flush`/`reseat` sequences against one Gateway's reactor rather than lost data. I did not audit `Gateway.reseat`/`federate` for re-entrancy, which is why confidence is low; if they are not re-entrant, this is worse than described.
    ✗ failure: An operator attaches one long-running quarantine pool C beneath two parents, P1 and P2, by adding it to both `quarantinePools` sets. `erase(X)` fans to P1 and P2 concurrently. Both complete their own purge and reseat at nearly the same instant, both read `seen` before either has added C, and both dispatch `C.eraseReplica(...)`. C then runs two overlapping `federate` + `flush` + `purge` + `reseat` sequences against the same reactor, and the guard that exists to make the walk finite has not actually excluded anything.
    → fix: Claim membership before awaiting rather than after: add each pool to `seen` synchronously at dispatch time — filter and add in the same non-async pass in both `eraseImpl` and `eraseReplicaImpl` — so a concurrently-dispatched sibling sees the claim. Have `eraseReplicaImpl` also check `seen.has(gw)` on entry rather than relying on the parent's filter. Add a rail attaching one pool gateway beneath two parents and assert it is visited exactly once.
